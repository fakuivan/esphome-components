Following example.yaml, we shoud build an esphome component similar to pylontech_rs485 that emulates a pylontech battery. pylontech_rs485 though is not standard compliant, so it currently does not work with my inverter. voltronic_vm_iv_queries_summary.md has info about real word communication with the inverter. The protocol docs (both transcriptio.md, transcribed from PDF) are contradictory on the structure of some of its packets, this is why I've also attatched some implementations (python-pylontech, esphome's pylontech component, pace2pylon and pylontech_rs485), it's your job to also compare conflicting documentation and arrive at a conclusion on what's the correct layout based on these implementations (we can start there). Another painpoint from pylontech_rs485 is that it directly used esphome analog sensors which are float based, we don't need to work with float if we already have integer values, which speeds up processing. These integer values will be returned from a lambda, and their units will need to match the unit in the specs (decivolt, millivolt, deci kelvin and so on) to again, minimize conversion time.

For now, let's only implement the `charge_discharge_management_info` and `analog_values` commands, as those are the only ones the inverter queries for.

Another important note is that command info encoders/parsers should be implemented in pylontech_rs485.hpp along with using the existing functions there to parse incoming requests and build outgoing responses. This means that you should not be doing info[x] = something in the esphome c++ files, rather using stuff like structs and info converters in the pylontech_rs485.hpp header file to construct the packets.
